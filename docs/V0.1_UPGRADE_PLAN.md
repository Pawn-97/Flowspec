# Flowspec v0.1 Upgrade Plan

## Status

Design boundary confirmed. Do not implement until this plan is reviewed.

## Goals

- Make downstream prototype handoff implementation-ready without prescribing a design system.
- Reduce downstream reading surface to one source-of-truth JSON plus one handoff Markdown.
- Default generated specs to the selected flow's happy path, with guardrails retained but not expanded.
- Extract candidate flows at independently prototypeable user-chain granularity.
- Prepare an admin portal skeleton draft for review before it becomes a default.

## Non-Goals

- Do not generate HTML, CSS, JavaScript, Figma, design tokens, backend integrations, external API calls, or project-management tasks.
- Do not inspect downstream prototype code in v0.1.
- Do not keep historical artifact snapshots in the internal audit directory.
- Do not hard-code design system install commands, package managers, framework setup, or component-library setup commands.

## Confirmed Artifact Layout

```text
flow-spec/
├── ux-flow-spec.json
├── handoff/
│   ├── ux-flow-spec.json
│   └── prototype-handoff.md
└── _internal/
    ├── flow-candidates.json
    ├── flow-index.md
    ├── flow-prep.json
    ├── flow-prep.md
    ├── ux-flow-spec.md
    ├── traceability.md
    ├── validation-report.json
    ├── ux-flow-spec-review.json
    ├── ux-flow-spec-review.md
    └── revision-log.md
```

`flow-spec/ux-flow-spec.json` remains the canonical working source of truth. `/ux-spec:handoff-prototype` publishes a copy to `flow-spec/handoff/ux-flow-spec.json`.

## Source JSON Changes

Add these top-level sections to every `ux-flow-spec.json`:

- `path_scope` — default branch coverage, included guardrails, excluded paths, and expansion policy.
- `prototype_surface` — skeleton status and design-system-agnostic implementation guidance.
- `artifact_contract` — downstream handoff files and internal audit directory contract.
- `acceptance_contract` — downstream implementation evidence requirements and acceptance checks.

Each screen must include `path_role`:

- `happy_path_step`
- `guardrail_state`
- `requested_expansion`
- `out_of_scope_reference`

Each concrete component must include `implementation_target`:

```json
{
  "strategy": "compose",
  "target_role": "editable data table",
  "interaction_shape": "row-level validation with recovery actions",
  "notes": "Optional guidance"
}
```

`strategy` is limited to `reuse_existing`, `compose`, or `custom`. FlowSpec must describe UI intent without requiring a named design system or component library.

## Handoff Markdown

`prototype-handoff.md` must be rendered deterministically from `ux-flow-spec.json`; no agent polishing is allowed.

Required sections:

- `Read This First`
- `Path Scope`
- `Prototype Surface`
- `Screens`
- `Agent Instructions`
- `Implementation Evidence Required`
- `Acceptance Checklist`
- `Evidence Summary`

`prototype-agent-prompt.md`, `prototype-brief.md`, and `prototype-brief.draft.md` are no longer default outputs. Their useful instructions move into `prototype-handoff.md`.

## Prototype Implementation Contract

Downstream prototype agents may use any suitable design system, component library, or local component stack.

Downstream agents must report implementation evidence:

- FlowSpec component id
- Concrete UI implementation mapping
- Any material deviation from the requested `implementation_target`

## Acceptance Checks

Use fixed check identifiers, with optional notes or additional text checks only when needed:

- `every_happy_path_step_screen_is_implemented`
- `every_required_component_has_mapping`
- `design_system_choice_is_not_prescribed`
- `implementation_deviations_are_reported`
- `excluded_paths_not_implemented`
- `no_backend_or_api_integration`

## Flow Extraction

Candidate flows should represent independently prototypeable user chains, not individual screens or entire product areas.

Each candidate should declare:

- entry
- success exit
- default path scope
- included guardrails
- excluded paths
- source trace

Default path scope is `happy_path`. Alternate and recovery paths require explicit follow-up expansion.

## Admin Portal Skeleton

Create `docs/skeletons/admin-portal.md` later as a reviewable draft, not in this planning pass.

The Admin portal keyscreen Figma link is a temporary skeleton extraction source only. After the skeleton is documented and approved, future FlowSpecs should not depend on the Figma link.

Skeleton status values:

- `none`
- `draft_pending_user_approval`
- `approved_default`

When status is `draft_pending_user_approval`, the skeleton is reference-only and must not be treated as an approved default.

## Workflow Changes

- `/ux-spec:extract-flows` writes candidate files to `flow-spec/_internal/` and prompts the user to review `flow-index.md`.
- `/ux-spec:prepare` writes prep files to `flow-spec/_internal/` and prompts the user to review `flow-prep.md`.
- `/ux-spec:generate` produces only `flow-spec/ux-flow-spec.json`.
- `/ux-spec:validate` and `/ux-spec:review` produce internal audit artifacts.
- `/ux-spec:handoff-prototype` checks gates and refreshes only `flow-spec/handoff/ux-flow-spec.json` and `flow-spec/handoff/prototype-handoff.md`.

## Implementation Phases

1. Update schemas for `path_scope`, `prototype_surface`, `artifact_contract`, `acceptance_contract`, `path_role`, and component-level `implementation_target`.
2. Update expected golden sample JSON to include the new contracts and per-component implementation targets.
3. Move generated internal artifacts under `flow-spec/_internal/`.
4. Replace prototype brief/prompt generation with deterministic `prototype-handoff.md` rendering.
5. Update command docs and `README.md` to describe the new artifact layout and workflow responsibilities.
6. Add or update tests for schema validation, artifact placement, deterministic handoff rendering, path roles, implementation targets, design-system freedom, and acceptance contract rendering.
7. After schema/workflow implementation, extract the admin portal skeleton from the Figma source and create `docs/skeletons/admin-portal.md` for review.

## Test Coverage To Add Or Update

- Schema accepts new top-level contracts and rejects missing required contract fields.
- `path_role` is required and limited to the four approved values.
- Default happy-path specs do not include unrequested expanded paths.
- Every concrete component has `implementation_target`.
- Every implementation target describes a `target_role`.
- `acceptance_contract.checks` uses approved fixed identifiers.
- Handoff command creates only `flow-spec/handoff/ux-flow-spec.json` and `flow-spec/handoff/prototype-handoff.md`.
- Internal artifacts are written under `flow-spec/_internal/`.
- `prototype-handoff.md` is rendered only from JSON content.
